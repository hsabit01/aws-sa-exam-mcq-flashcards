## Elastic Storage Block (ESB)

### What is it?
This topic is really **Amazon Elastic Block Store (EBS)**.

EBS is **block storage** for EC2.  
Think of it like a virtual hard drive for one server.

It is **persistent storage**, so the data stays even if you stop the instance.

### How it works?
You create an EBS volume in an **Availability Zone**.

Then you attach it to an EC2 instance in the **same AZ**.

The instance reads and writes blocks of data to that volume, just like using a disk.

You can also resize or tune some EBS volumes without rebuilding everything.

### Use Case
A company runs a MySQL database on EC2.

The database needs storage that stays after reboot and gives steady performance.

EBS is a strong answer because it is persistent block storage for EC2.

### Exam Tip
If the question says **EC2 needs a persistent disk**, think **EBS**.

If it sounds like **one server / one OS / one database disk**, EBS is usually the right answer.

Trap: EBS is **not shared file storage** across many instances.  
If many Linux instances need the same files at the same time, look at **EFS** instead.

## EBS Snapshot

### What is it?
An EBS snapshot is a **point-in-time backup** of an EBS volume.

It is **incremental**, so after the first snapshot, AWS stores only changed blocks.

### How it works?
You take a snapshot of a volume.

Later, you can create a **new EBS volume** from that snapshot.

This is useful for backup, restore, migration, and disaster recovery.

### Use Case
A team wants daily backups of an EC2 database volume.

If the volume is lost, they can restore a new EBS volume from the latest snapshot.

### Exam Tip
If the question says **backup an EBS volume**, **restore later**, or **copy a volume**, think **snapshot**.

A snapshot is about a **volume backup**.  
An AMI is about launching a **whole EC2 server template**.

Trap: snapshot is not shared live storage. It is backup/restore storage.

## EBS Snapshot Features

### What is it?
These are extra snapshot capabilities that help with backup, recovery, retention, and compliance.

For the exam, the most useful features are **Fast Snapshot Restore**, **Snapshot Archive**, **Recycle Bin**, **Snapshot Lock**, and **automation with Data Lifecycle Manager**.

### How it works?
**Fast Snapshot Restore** makes restored volumes fully initialized right away.

**Snapshot Archive** is a lower-cost tier for long-term retention.

**Recycle Bin** helps recover deleted snapshots.

**Snapshot Lock** protects snapshots from deletion and can support WORM-style retention.

**Data Lifecycle Manager** automates snapshot creation, retention, archive, and some advanced settings.

### Use Case
A company needs daily backups, long retention, fast disaster recovery, and protection from accidental deletion.

They can automate snapshots with DLM, archive old snapshots, use Recycle Bin for recovery, and enable Fast Snapshot Restore for low RTO.

### Exam Tip
Look for these clue phrases:

**“faster restore”** → Fast Snapshot Restore  
**“long-term low-cost retention”** → Snapshot Archive  
**“recover deleted snapshot”** → Recycle Bin  
**“cannot delete due to compliance”** → Snapshot Lock  
**“automated schedules and retention”** → Data Lifecycle Manager

Trap: snapshots are still backups. They do not replace a shared file system like EFS.

## Amazon Machine Image (AMI)

### What is it?
An AMI is a **template** used to launch an EC2 instance.

It includes the software needed to boot the server, plus block device mapping.

### How it works?
You choose an AMI when launching EC2.

AWS uses that AMI to create the instance root volume and start the machine.

An AMI can be based on snapshots of the root volume.

### Use Case
A company wants ten identical web servers.

They create a custom AMI with the OS, patches, app code, and settings already installed.

Then they launch many EC2 instances from it.

### Exam Tip
If the question says **“launch identical EC2 instances quickly”**, think **AMI**.

If it says **“backup one disk”**, think **snapshot**.

A very common exam distinction is:

**Snapshot** = backup of a volume  
**AMI** = template for launching an instance

Trap: AMI is not a shared storage service.

## EBS Volume Types

### What is it?
EBS has different volume types for different workloads.

The main exam types are **gp3**, **gp2**, **io1/io2**, **st1**, and **sc1**.

### How it works?
**gp3** is the common default for general use.  
It gives a good balance of price and performance.

**gp2** is the older general-purpose SSD option.

**io1/io2** are for very high IOPS and critical workloads like databases.

**st1** is HDD for large, sequential throughput workloads.

**sc1** is cheaper HDD for cold, infrequently accessed data.

### Use Case
A web app boot volume usually fits **gp3**.

A critical database with very high IOPS may need **io2**.

A log-processing or big sequential analytics workload may fit **st1**.

### Exam Tip
If the workload says **database**, **high IOPS**, **critical**, or **low latency**, think **io1/io2**.

If it says **general-purpose SSD**, think **gp3** first.

If it says **large sequential throughput**, think **st1**.

If it says **lowest-cost HDD for cold data**, think **sc1**.

Trap: **st1 and sc1 cannot be boot volumes**.  
Trap: HDD volumes are not a good fit for small random I/O.

## EBS Multi-Attach

### What is it?
EBS Multi-Attach lets **one EBS volume** attach to **multiple EC2 instances** at the same time.

### How it works?
You enable Multi-Attach on a supported EBS volume.

That volume can then attach to multiple supported instances in the **same Availability Zone**.

All attached instances can read and write to the shared volume.

### Use Case
A clustered application in one AZ needs shared block storage with application-level coordination.

Multi-Attach can help with higher application availability in those special designs.

### Exam Tip
The main clues are:

**“same EBS volume attached to multiple EC2 instances”**  
**“same Availability Zone”**  
**“shared block storage”**

Trap: this is **not** general shared file storage for many servers.  
For most “many servers need shared files” questions, the answer is **EFS**, not Multi-Attach.

Trap: Multi-Attach supports **io1/io2** volumes and the instances must be in the **same AZ**.

Trap: your app or cluster-aware file system must handle concurrent writes correctly.

## Elastic File System (EFS)

### What is it?
EFS is a **managed NFS file system** for AWS.

It is designed for **shared file storage** and can be mounted by many compute resources.

### How it works?
You create an EFS file system.

Then you create **mount targets** in your VPC so clients can connect.

EC2 instances can mount it over **NFS** and share the same files.

EFS can scale automatically as files are added or removed.

### Use Case
A fleet of Linux web servers needs a shared uploads folder.

All servers mount the same EFS file system and see the same files.

### Exam Tip
If the question says **shared file system**, **Linux**, **NFS**, or **multiple EC2 instances need the same files**, think **EFS**.

EFS is usually about **managed**, **elastic**, **multi-instance** file storage.

Trap: EFS is file storage, not block storage.  
If the question is about **boot disks**, **database block storage**, or **one server disk**, think EBS instead.

## EFS Use Cases and Access

### What is it?
This topic is about **where EFS fits** and **how clients connect**.

EFS is best when many systems need the same file data.

### How it works?
Access happens through **mount targets** in the VPC.

You usually create mount targets across AZs for a Regional EFS file system.

Applications connect over **NFS port 2049**.

You can use **EFS Access Points** to give an app its own path and permissions.

EFS can also be reached from on-premises if network connectivity exists to the VPC, such as through Direct Connect.

### Use Case
A container platform needs shared persistent files for many app tasks.

Each task uses EFS, and access points give each app a cleaner, safer entry point.

### Exam Tip
Look for these clues:

**“shared Linux files across AZs”**  
**“NFS”**  
**“mount target”**  
**“access point”**  
**“POSIX permissions”**

Trap: **security groups are applied at the mount target**, not at the access point.

Trap: for Windows-style file shares, EFS is usually not the exam answer.

## EFS Performance and Throughput Modes

### What is it?
EFS lets you choose **performance mode** and **throughput mode**.

These settings help balance latency, scale, and cost.

### How it works?
For **performance mode**, the key choices are:

**General Purpose** for low-latency workloads.  
**Max I/O** for very high scale and concurrency.

For **throughput mode**, the choices are:

**Elastic** when you want throughput to scale automatically with usage.  
**Provisioned** when you need a set throughput level.  
**Bursting** when throughput grows based on file system size and burst credits.

### Use Case
A CMS or shared web content system usually fits **General Purpose**.

A very large analytics or media workload with many parallel operations may fit **Max I/O**.

A spiky workload with changing traffic often fits **Elastic throughput**.

### Exam Tip
If the question says **low latency** or **most workloads**, think **General Purpose**.

If it says **very high parallelism / thousands of clients / analytics scale**, think **Max I/O**.

If it says **unpredictable throughput** and you want less manual tuning, think **Elastic throughput**.

Trap: students often confuse **performance mode** with **throughput mode**.  
They are different settings.

## EFS Storage Classes and Lifecycle Tiers

### What is it?
EFS has storage options to lower cost as files become less active.

The main ideas are **Regional vs One Zone** and **Standard vs IA vs Archive**.

### How it works?
**Regional** stores data across multiple AZs in a Region.  
This is the recommended option for higher resilience.

**One Zone** stores data in one AZ and is cheaper, but it has less resilience.

With lifecycle policies, files can move from **Standard** to **Infrequent Access (IA)** and then to **Archive** based on how long they have not been accessed.

When data in IA or Archive is written again, writes first go to **Standard**, then can move later.

### Use Case
A company has millions of files.

Recent files must stay fast, but old files should become cheaper automatically.

EFS lifecycle policies handle that without changing the app.

### Exam Tip
If the question says **shared files**, **automatic cost optimization**, and **older files accessed less often**, think **EFS lifecycle + IA/Archive**.

If the question says **highest resilience**, choose **Regional**.

If it says **lower cost is more important than Multi-AZ resilience**, **One Zone** may be the answer.

Trap: **Archive** is not for frequently accessed data.  
Trap: **One Zone** is cheaper but less resilient than Regional.

## EBS vs EFS vs Instance Store

### What is it?
These are three very different EC2 storage answers.

This comparison is one of the most important SAA exam topics.

### How it works?
**EBS** is persistent **block storage** for EC2.  
Best for one instance disk, boot volumes, and databases.

**EFS** is managed **file storage** shared over NFS.  
Best when many Linux systems need the same files.

**Instance Store** is **temporary local block storage** physically attached to the host.  
It is very fast, but the data is lost if the instance is terminated or the host fails.

### Use Case
Use **EBS** for an EC2 database server.

Use **EFS** for shared content across many app servers.

Use **Instance Store** for cache, buffer, scratch data, or temporary processing.

### Exam Tip
If the exam says:

**persistent disk for one EC2** → EBS  
**shared file storage across many EC2s** → EFS  
**very fast temporary storage / cache / buffer / scratch** → Instance Store

Trap: students choose Instance Store for important data because it is fast.  
That is usually wrong because it is **ephemeral**.

Trap: students choose EBS when the question clearly needs **many instances sharing the same files**.  
That is usually an EFS question.
